The Big Idea

What is RAG — in plain English?

Imagine you hired a very smart employee who reads fast. Before answering your question, they go to a filing cabinet, pull out the most relevant pages, read them, and then give you a precise answer based on what they just read. That is RAG.

🤖

Without RAG — Pure AI

A regular AI only knows what it was trained on months or years ago. Ask it about your internal policy document? It has no idea. It will guess — and often get it wrong. This is called "hallucination".

📚

The Filing Cabinet Analogy

RAG gives the AI a filing cabinet — your own documents. When you ask a question, the system first fetches the most relevant pages from your cabinet, then asks the AI to answer using only those pages.

🎯

Why RAG wins

The AI's answer is grounded in your actual data. No hallucinations from outdated training. You can update the filing cabinet any time by uploading new documents. The AI just reads; your data drives the answers.

⚙️

This Project in One Line

Upload a PDF → chop it into small pieces → turn each piece into numbers → store them → when asked a question, find the most similar pieces → feed them to an AI → get a smart, sourced answer.

💡

The core insight

RAG = Search engine + AI language model working together. The search engine finds the right pages; the AI reads them and talks to you.

Vocabulary

Four concepts you must know

These four ideas are the engine of every RAG system. Understand these and the code will make complete sense.

01

Chunking ✂️

A 100-page PDF is too big to process at once. We slice it into small overlapping pieces called chunks — like cutting a book into sticky notes. Each chunk is roughly 1000 characters with a 200-character overlap with its neighbor so context is never cut off mid-sentence.

02

Embeddings 🔢

An embedding is a list of ~1024 numbers that represent the meaning of a text chunk. Think of it as GPS coordinates for meaning — two chunks that mean similar things get coordinates that are close to each other in a giant mathematical space. AWS Bedrock's Titan model generates these numbers.

03

Vector Search 🔍

When you ask a question, the question is also turned into embedding numbers. Then Elasticsearch finds the stored chunks whose numbers are closest (using cosine similarity) to the question's numbers. "Closest numbers" = "most relevant text". This is how the system finds the right pages without keyword matching.

04

LLM Answer Generation 💬

The top matching chunks become the context. This context plus your question are sent to an AI language model (Amazon Nova Lite via AWS Bedrock). The model is instructed: "Only answer using this context." The result is a precise, document-grounded reply.

Why overlap when chunking?

If a sentence like "The CEO resigned because of…" falls right at the boundary of two chunks, one chunk loses the beginning and the other loses the ending. A 200-character overlap makes sure important sentences are fully captured in at least one chunk.

Why does the prompt say "only use the context"?

Without that instruction, the LLM might mix in knowledge from its training data and give you an answer that sounds right but isn't from your document. The strict prompt keeps the model honest — if the answer isn't in your PDF, it says so.

The front door — starts the web server and wires routes together

This is the first file Node.js runs when you type node server.js. It creates an Express web server, applies two middleware layers (CORS for browser access, JSON body parsing), then connects the two route files to handle incoming HTTP requests.

Services — the worker layer

📄
services/pdfService.js
Reads a PDF file buffer and returns plain text
▼

A thin wrapper around the pdf-parse library. You give it a binary file buffer (the raw bytes of a PDF), and it hands back a plain string of all the text inside — no formatting, no images, just the words.

Why a separate file? Because if you ever want to swap pdf-parse for a different PDF library, you only change this one file. Everything else in the project keeps working.

services/chunkService.js
Slices a long text string into overlapping pieces called chunks
▼

Imagine you have a very long text. You can't feed it all to the embedding model at once (models have limits), and even if you could, searching across a single giant blob is inefficient. This service chops the text into manageable pieces.

The overlap is the clever part: each chunk shares 200 characters with the previous one. This ensures a sentence that spans the boundary of two chunks is fully represented in at least one of them, preventing context loss.

services/embeddingService.js
Converts any text string into a 1024-number vector using AWS Bedrock
▼

This is the translation layer — it turns human language into the mathematics that computers can compare. It calls Amazon Bedrock's Titan Embed Text v2 model, which returns an array of 1024 floating-point numbers that encode the semantic meaning of the text.

The same service is used for both storing document chunks and for embedding user questions. This symmetry is crucial — both must use the same model so the numbers are in the same mathematical space and comparisons work correctly.

services/indexService.js
Creates the Elasticsearch index with the correct schema — runs once before storing
▼

Before you can store anything in Elasticsearch, you need an index — think of it as a table in a database. This service creates the pdf_chunks index with a precise schema (called a mapping) that tells Elasticsearch the exact type of every field.

The most important field in the mapping is embedding — defined as a dense_vector with 1024 dimensions and cosine similarity. This is what enables the magical nearest-neighbor search. The service also checks if the index already exists so it never duplicates it.

services/ingestionService.js
Saves one chunk + its embedding vector into Elasticsearch
▼

The storage worker. Given a chunk object and its embedding array, it bundles them together into a document and saves it to the pdf_chunks index. Each document gets a deterministic ID in the format filename-chunkNumber so re-uploading the same file overwrites rather than duplicates.

🔍
services/retrievalService.js
Queries Elasticsearch with a vector and returns the most semantically similar chunks
▼

This is the search engine of the RAG system. It receives the question embedding (1024 numbers) and asks Elasticsearch to find the 5 stored chunks whose embedding vectors are most similar using the kNN (k-Nearest Neighbors) algorithm.

"Most similar vectors" = "chunks whose meaning is closest to the question's meaning". This is semantic search — not keyword matching. You could ask "What caused revenue decline?" and it would find a chunk that says "sales dropped due to supply chain issues" even without any word overlap.

100 (num_candidates)

💬
services/llmService.js
Sends context + question to Amazon Nova Lite and returns the AI-generated answer
▼

The brain of the system. It calls Amazon Nova Lite (a fast, cheap, capable language model on AWS Bedrock) with a carefully crafted prompt that instructs the model to answer only from the provided context.

The prompt has three parts: a system instruction (be a helpful assistant, only use the context), a context block (the joined text of the top-5 retrieved chunks), and the user question. If the answer isn't found in the context, the model is told to say so — this prevents hallucination.

AWS Bedrock

🏗️

Local vs Cloud — where does data live?

Elasticsearch runs locally on your machine (port 9200) — it's your vector database. AWS Bedrock is a cloud API — it's where the AI models live (Titan for embeddings, Nova Lite for answers). S3 is optional cloud storage for the original PDF files.
